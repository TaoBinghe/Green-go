# Sprint 2 需求说明

## 1. Sprint 目标

Sprint 2 的后端目标是在 Sprint 1 已具备基础注册、登录、预订与模拟支付能力的前提下，补齐订单闭环管理、车辆可用性联动、车辆位置数据支撑，以及管理端基础运营能力，使系统具备更完整的租赁管理能力。

本需求文档仅定义 Sprint 2 的后端范围。

## 2. 订单生命周期管理

### 2.1 订单状态

系统应支持以下订单状态：

- `PENDING`
- `ACTIVE`
- `COMPLETED`
- `CANCELLED`

系统应保证订单状态在后端持久化，并可用于订单查询、状态判断与业务流转。

### 2.2 激活订单

- 用户应能够激活属于自己的 `PENDING` 订单。
- 激活成功后，订单状态应更新为 `ACTIVE`。
- 若用户已存在其他未关闭订单，系统应拒绝激活并返回明确错误信息。

### 2.3 取消订单

- 用户应能够取消属于自己的 `PENDING` 订单。
- 取消成功后，订单状态应更新为 `CANCELLED`。
- `ACTIVE` 状态订单不允许直接取消。

### 2.4 结束订单

- 用户应能够结束属于自己的 `ACTIVE` 订单。
- 结束订单时，系统应执行模拟支付。
- 支付成功后，订单状态应更新为 `COMPLETED`。
- 系统应记录实际结束时间。
- 系统应返回更新后的订单信息及对应支付记录。

### 2.5 用户订单查询

- 用户应能够查询自己的订单列表。
- 查询结果应同时包含当前订单与历史订单。
- 订单列表应按创建时间倒序返回。

### 2.6 兼容状态切换接口

为兼容现有客户端流程，系统应提供订单状态切换接口：

- 支持在 `PENDING` 与 `ACTIVE` 之间切换
- 前端传入的 `ACTIVATED` 应在后端视为 `ACTIVE`
- 非法状态切换应被拒绝

## 3. 改租期与续租

### 3.1 修改待激活订单租期

- 用户应能够修改属于自己的 `PENDING` 订单租期。
- 修改租期时，系统应根据新的定价方案重新计算：
  - `pricingPlanId`
  - `endTime`
  - `totalCost`
- 非 `PENDING` 状态订单不允许修改租期。

### 3.2 续租进行中订单

- 用户应能够续租属于自己的 `ACTIVE` 订单。
- 续租时可选租期仍应沿用系统已有固定租期：
  - `HOUR_1`
  - `HOUR_4`
  - `DAY_1`
  - `WEEK_1`
- 续租成功后，系统应在原结束时间基础上延长 `endTime`。
- 续租成功后，系统应将新增费用累加到 `totalCost`。

### 3.3 续租可用性检查

- 在确认续租前，系统应检查对应车辆在新增延长期内是否仍可用。
- 若同一车辆存在其他开放订单与新增延长期冲突，则系统应拒绝续租并返回明确错误信息。

### 3.4 支付规则

- Sprint 2 的后端支付规则应为“结束订单时支付”。
- 同一订单只允许存在一条支付记录。
- 续租不应单独生成新的支付记录，而应在订单结束时统一结算。

## 4. 车辆状态与可用性联动

### 4.1 车辆状态模型

系统应使用以下车辆状态：

- `AVAILABLE`
- `UNAVAILABLE`

### 4.2 订单与车辆状态联动

- 创建订单前，系统必须重新校验车辆当前是否可用。
- 创建订单成功后，车辆状态应更新为 `UNAVAILABLE`。
- 取消 `PENDING` 订单后，车辆状态应恢复为 `AVAILABLE`。
- 完成 `ACTIVE` 订单并支付后，车辆状态应恢复为 `AVAILABLE`。

### 4.3 可用性校验

- 若车辆不存在，系统应拒绝下单。
- 若车辆当前为不可用状态，系统应拒绝下单。
- 若车辆已被其他开放订单占用，系统应拒绝下单。

## 5. 车辆列表与位置数据

### 5.1 车辆列表

系统应提供车辆列表能力，输出每辆车至少包括：

- scooter ID
- scooter code
- 当前状态
- 位置名称
- 经度
- 纬度

### 5.2 位置数据支撑

- 后端应保存车辆位置名称与坐标信息。
- 车辆列表返回结果应可直接支撑列表展示与地图类页面使用。

### 5.3 管理端坐标解析

- 管理端应支持根据经纬度解析位置名称。
- 若坐标无法解析为有效地址，系统应返回失败信息。

## 6. 管理端能力

### 6.1 管理员鉴权

- 系统应支持管理员登录。
- 管理端接口应要求当前用户角色为 `MANAGER`。
- 非管理员访问管理端接口时，系统应返回 `"Forbidden: admin only"`。

### 6.2 车辆管理

管理端应支持：

- 查看车辆列表
- 新增车辆
- 更新车辆
- 删除车辆

并满足以下约束：

- 车辆编号必须唯一
- 经纬度必须合法
- 新增与更新时应校验坐标是否能解析出有效地址

### 6.3 定价方案管理

管理端应支持定价方案 CRUD：

- 获取全部定价方案
- 获取单个定价方案
- 新增定价方案
- 更新定价方案
- 删除定价方案

并满足以下约束：

- 租期代码不能为空
- 价格必须大于 0
- 租期代码必须唯一
- 已被订单使用的定价方案不能删除

## 7. 管理端周收入统计

### 7.1 统计范围

管理端应支持查看最近一周收入统计。

统计口径应满足：

- 仅统计支付成功记录
- 按支付时间统计最近 7 天
- 按固定租期类型统计：
  - `HOUR_1`
  - `HOUR_4`
  - `DAY_1`
  - `WEEK_1`

### 7.2 统计维度

每个租期类型至少应返回：

- 订单数量
- 总收入

系统还应返回：

- `windowStart`
- `windowEnd`
- `mostPopularHirePeriod`

### 7.3 空数据处理

- 即使某一租期在统计窗口内没有数据，系统仍应返回该租期对应的 0 值统计结果。

## 8. 测试要求

Sprint 2 后端应补充自动化测试，至少覆盖以下能力：

- 订单激活
- 订单取消
- 修改租期
- 订单续租
- 续租冲突校验
- 完成订单与支付联动
- 车辆坐标与地址处理
- 周收入统计聚合逻辑

## 9. Sprint 2 后端交付结果

Sprint 2 后端交付应满足以下要求：

- 系统具备完整的订单状态流转能力
- 系统具备待激活改租期与进行中续租能力
- 系统具备车辆可用性校验与车辆状态联动能力
- 系统具备车辆位置数据输出能力
- 管理端具备车辆管理、定价方案管理与周收入统计能力
- 核心后端流程具备基础自动化测试支撑
